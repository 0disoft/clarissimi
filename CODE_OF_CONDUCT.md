# Code of Conduct

- Status: Draft
- Owner: Maintainers

## Purpose

Clarissimi exists to recognize open-source contribution without turning contributors into
a public scoreboard. The same standard applies to project participation: discussion should
help people leave clearer evidence, safer decisions, and better software behind.

## Expected Behavior

- Be direct, specific, and kind.
- Critique ideas, code, docs, and decisions without attacking people.
- Assume public comments may be read by new contributors trying to decide whether this
  project is safe to join.
- Keep recognition language evidence-based and maintainer-approved.
- Respect that bug reports, reproduction cases, tests, docs, design feedback, security
  reports, accessibility work, translations, and release validation can all be meaningful
  contributions.

## Unacceptable Behavior

- Harassment, insults, threats, or sexualized language.
- Public ranking, shaming, or contributor score speculation.
- Pressure to disclose private identity, employment, contact details, or security-sensitive
  information.
- Posting secrets, tokens, exploit details, private keys, private emails, or unreleased
  vulnerability details in public issues, pull requests, discussions, logs, or examples.
- Repeatedly derailing maintainer decisions after a boundary has been stated.

## Recognition-Specific Norms

Clarissimi should not be used to decide whether a person is good, valuable, senior, hireable,
or better than another contributor. It should describe what a maintainer-approved contribution
changed, what evidence supports that description, and how the project wants to remember it.

Do not submit changes that add public leaderboards, public total scores, contributor tiers, or
ranking language unless the product contract is explicitly changed first.

## Reporting and Moderation

For ordinary conduct concerns, use the repository moderation path made available by the
maintainers. If no private conduct channel is published yet, do not pretend one exists in
project documentation. Maintainers should handle public issues, pull requests, and discussions
using GitHub moderation tools and may hide, edit, lock, or remove content that violates this
code.

For security-sensitive reports, follow SECURITY.md and use GitHub Security Advisories when
available. Do not open a public issue with exploit details or secrets.

Maintainers may take one or more of these actions:

- ask for edits or a change in tone
- remove harmful content
- close, lock, or hide a thread
- block repeat abuse
- decline recognition text that is speculative, ranking-oriented, or unsupported by evidence

## Maintainer Standard

Maintainers are expected to apply this code consistently, especially when a contribution is
controversial, security-sensitive, or emotionally loaded. Recognition should remain grounded in
repository evidence and project policy, not popularity or personal preference.
